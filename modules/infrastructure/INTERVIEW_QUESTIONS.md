# Вопросы для собеседования

36 вопросов по инфраструктурным темам для backend-разработчика.

---

## Docker (Q1–Q8)

### Q1: Чем отличается `COPY` от `ADD` в Dockerfile?
**A:** `COPY` просто копирует файлы из контекста сборки в образ — это предпочтительный вариант для большинства случаев. `ADD` дополнительно умеет распаковывать tar-архивы и загружать файлы по URL. Именно из-за неявного поведения `ADD` рекомендуется использовать только когда нужна распаковка; для простого копирования всегда предпочитайте `COPY`.

### Q2: Что такое слой образа? Почему порядок инструкций в Dockerfile важен?
**A:** Каждая инструкция в Dockerfile создаёт новый слой — неизменяемый diff файловой системы. Docker кэширует слои: если слой не изменился, он берётся из кэша. Если поставить `COPY . .` перед `RUN mvn dependency:go-offline`, то любое изменение исходного кода инвалидирует кэш зависимостей и Maven скачивает их заново. Правильный порядок: сначала `pom.xml` + загрузка зависимостей, потом исходный код.

### Q3: В чём смысл многоэтапной (multi-stage) сборки?
**A:** Multi-stage позволяет разделить build-окружение и runtime-окружение. В первом stage используется образ с JDK и Maven для компиляции. Во второй stage копируется только собранный `*.jar` поверх минимального JRE-образа (~70 МБ вместо ~500 МБ). В финальный образ не попадают Maven, исходный код и промежуточные артефакты — меньше размер, меньше attack surface.

### Q4: Чем отличается `ENTRYPOINT` от `CMD`?
**A:** `ENTRYPOINT` задаёт исполняемый файл контейнера, который нельзя переопределить при запуске без флага `--entrypoint`. `CMD` — аргументы по умолчанию, которые легко заменяются через `docker run <image> <args>`. Для приложений рекомендуется `ENTRYPOINT ["java", "-jar", "app.jar"]` с `CMD` для дефолтных флагов. Форма exec (`["java", "-jar"]`) предпочтительнее shell-формы (`java -jar`) — PID 1 получает сигналы напрямую.

### Q5: Как контейнеры общаются друг с другом в docker-compose?
**A:** Контейнеры в одной compose-сети могут обращаться друг к другу по имени сервиса как по DNS-имени. Например, сервис `app` обращается к `postgres:5432` — имя `postgres` резолвится внутри сети автоматически. Контейнеры в разных сетях изолированы и не видят друг друга, если только сеть не объявлена как `external` или сервис не подключён к обеим сетям.

### Q6: Зачем запускать контейнер от non-root user?
**A:** По умолчанию процессы в контейнере запускаются от root. Если злоумышленник получает RCE в приложении, он получает root внутри контейнера. При mount хоста или container escape это может означать root на хосте. Non-root пользователь ограничивает blast radius: нет доступа к системным файлам, нет возможности открыть привилегированные порты (<1024). Создаётся через `RUN useradd -r appuser && USER appuser`.

### Q7: Чем именованный volume (named volume) отличается от bind mount?
**A:** Bind mount монтирует конкретный путь на хосте (`-v /host/path:/container/path`) — зависит от файловой системы хоста, удобен для разработки. Named volume (`-v mydata:/container/path`) управляется Docker: хранится в `/var/lib/docker/volumes/`, переносим между хостами через `docker volume export`, не зависит от структуры директорий хоста. Для production данных (база данных, файлы приложения) используйте named volumes.

### Q8: Что такое `HEALTHCHECK` и зачем он нужен?
**A:** `HEALTHCHECK` задаёт команду, которую Docker периодически запускает для проверки состояния контейнера. Если проверка проваливается N раз подряд, контейнер помечается как `unhealthy`. В docker-compose это позволяет `depends_on` с `condition: service_healthy` ждать реальной готовности сервиса, а не просто факта его запуска. Без HEALTHCHECK `depends_on` лишь гарантирует порядок старта, но не готовность.

## Kubernetes (Q9–Q16)

### Q9: В чём разница между Pod и Deployment?
**A:** Pod — минимальная единица деплоя в Kubernetes, группа контейнеров с общей сетью и хранилищем. Pod смертен: при падении он не восстанавливается автоматически. Deployment управляет ReplicaSet, который обеспечивает нужное количество живых Pod. Deployment умеет делать rolling update (постепенную замену Pod новой версией) и rollback. На практике Pod в изоляции почти не создают — только через Deployment или другие контроллеры.

### Q10: Как Kubernetes узнаёт, что Pod готов принимать трафик?
**A:** Через `readinessProbe`. Пока проверка не проходит, Pod не добавляется в Endpoints сервиса и трафик на него не идёт. `livenessProbe` — другая проверка: если она падает, kubelet перезапускает контейнер. Важно: не давать им один endpoint, иначе кратковременная проблема (например, высокая нагрузка) вызовет перезапуск вместо того, чтобы просто временно вывести Pod из балансировки.

### Q11: Чем ConfigMap отличается от Secret?
**A:** ConfigMap — для нечувствительной конфигурации (URL сервисов, feature flags). Secret — для чувствительных данных (пароли, токены). Оба монтируются как env variables или файлы. Ключевые различия: Secret хранятся в etcd зашифрованными (при включённом encryption at rest), доступ контролируется RBAC отдельно от ConfigMap. В коде разницы нет — Secret такие же переменные окружения. Важно не хранить Secret в git в plain text.

### Q12: Что происходит, если `livenessProbe` провалилась?
**A:** Kubelet перезапускает контейнер (не Pod целиком). Если `restartPolicy: Always` (по умолчанию для Deployment), контейнер перезапустится с экспоненциальным backoff (10s, 20s, 40s... до 5 минут). Если `livenessProbe` настроена слишком агрессивно (маленький `timeoutSeconds` или `failureThreshold`), приложение с медленным стартом будет постоянно перезапускаться — это crashloop. Решение: `startupProbe` для защиты медленного старта.

### Q13: Как работает HPA? Что нужно задать обязательно?
**A:** HPA (Horizontal Pod Autoscaler) периодически (каждые 15 секунд) смотрит на текущее потребление ресурсов Pod и сравнивает с целевым значением. Если CPU > 70% → добавляет реплики, если CPU < 70% → убирает (с кулдауном). Обязательно нужны `resources.requests` в Pod spec — без них HPA не знает относительно чего считать проценты. `requests` ≠ `limits`: requests определяет расчёт для HPA и scheduler, limits — жёсткий потолок.

### Q14: Чем ClusterIP отличается от NodePort?
**A:** ClusterIP — виртуальный IP внутри кластера, доступен только изнутри (Pod ↔ Pod). NodePort — открывает порт (30000-32767) на каждой ноде кластера, делая сервис доступным снаружи через `<node-ip>:<node-port>`. LoadBalancer — NodePort + создание облачного балансировщика. Для внутреннего взаимодействия микросервисов всегда ClusterIP. NodePort/LoadBalancer — только для внешних запросов. DNS имя сервиса внутри кластера: `<service>.<namespace>.svc.cluster.local`.

### Q15: Что такое resource requests vs limits? Какие QoS классы есть?
**A:** `requests` — гарантия: scheduler размещает Pod на ноде, где есть столько ресурсов. `limits` — потолок: при превышении CPU контейнер throttle-ится, при превышении памяти — OOMKilled. QoS классы: **Guaranteed** (requests == limits для всех контейнеров) — Pod выселяется последним; **Burstable** (requests < limits) — средний приоритет; **BestEffort** (нет requests/limits) — выселяется первым при нехватке ресурсов.

### Q16: Зачем нужны namespaces?
**A:** Namespaces — виртуальные кластеры внутри одного физического кластера. Обеспечивают: изоляцию ресурсов (два сервиса с именем `api` в разных namespace не конфликтуют), контроль доступа через RBAC (команда А видит только свой namespace), ResourceQuota (ограничение CPU/memory для namespace), LimitRange (дефолтные requests/limits). Типичное деление: `production`, `staging`, `monitoring`, `infra`.

## Helm (Q17–Q20)

### Q17: Что такое Helm release? Чем отличается от chart?
**A:** Chart — пакет с шаблонами Kubernetes манифестов и `values.yaml`. Release — конкретная установка chart в кластер с определёнными values. Один chart можно установить несколько раз под разными именами release (например, `myapp-prod` и `myapp-staging`). Helm хранит историю releases в Kubernetes Secrets, что позволяет делать `helm rollback` к предыдущей версии.

### Q18: Как передать значения в chart при установке?
**A:** Три способа: `--set key=value` (CLI, для единичных значений), `--values custom.yaml` (файл, для многих значений), `--set-string` (форсировать тип string). При `helm upgrade` значения из предыдущего release не сохраняются автоматически — нужно `--reuse-values` или явно передать все нужные. В CI/CD обычно используют `-f values-prod.yaml` для разных окружений.

### Q19: Зачем нужны Helm hooks? Приведи пример.
**A:** Hooks позволяют запускать Jobs в определённые моменты lifecycle release: `pre-install` (до создания ресурсов), `post-install`, `pre-upgrade`, `post-upgrade`, `pre-delete`. Типичный пример: Job с DB миграцией как `pre-upgrade` hook — Helm ждёт его завершения перед деплоем новых Pod. Без hooks пришлось бы вручную координировать миграцию и деплой. Ресурсы hooks не управляются как обычные release ресурсы: нужен `hook-delete-policy` чтобы их убирать.

### Q20: Что делает `helm rollback`?
**A:** Откатывает release к предыдущей ревизии (или к указанной). Helm берёт сохранённые манифесты предыдущей ревизии из Kubernetes Secret и применяет их. Важно: rollback создаёт новую ревизию (не удаляет текущую). Hooks `pre-rollback` и `post-rollback` тоже выполняются. Ограничение: если между деплоями была DB миграция, rollback приложения не откатывает схему БД — нужна отдельная down-миграция.

## Observability (Q21–Q24)

### Q21: В чём разница между метриками, логами и трейсами?
**A:** Метрики — агрегированные числовые показатели во времени (RPS, error rate, latency p99). Дёшевы для хранения и запросов, но теряют детали. Логи — записи конкретных событий с контекстом. Дороги при большом объёме, но незаменимы для отладки. Трейсы — путь запроса через несколько сервисов (распределённая трассировка). Показывают где именно тратится время в цепочке. На практике используются вместе: метрика сигнализирует о проблеме, лог объясняет что произошло, трейс показывает где.

### Q22: Что такое SLI, SLO, SLA? Дай пример для API сервиса.
**A:** SLI (Service Level Indicator) — измеримый показатель качества: "доля запросов, выполненных за < 200ms". SLO (Service Level Objective) — целевое значение SLI: "99% запросов выполняются за < 200ms". SLA (Service Level Agreement) — контракт с пользователем с последствиями нарушения (скидка, штраф). SLO строже SLA — чтобы иметь запас до нарушения контракта. Пример: SLI = `rate(http_requests_total{status!~"5.."}[5m]) / rate(http_requests_total[5m])`, SLO = 99.9% success rate.

### Q23: Что такое error budget?
**A:** Error budget = 1 - SLO. Если SLO = 99.9%, то error budget = 0.1% = ~43 минуты недоступности в месяц. Пока budget не исчерпан, команда может деплоить новые фичи. Когда budget исчерпан — фокус переключается на надёжность, новые деплои замораживаются. Это механизм баланса между скоростью разработки и стабильностью. Для разработчика важно понимать: каждый deploy несёт риск, который измеряется в минутах error budget.

### Q24: Как связать лог, метрику и трейс в production?
**A:** Через correlation ID — уникальный идентификатор запроса, который прокидывается через все три системы. В логах он присутствует как поле (`trace_id`), в метриках — реже (высокая кардинальность), в трейсах — это trace ID. Современный стандарт: W3C TraceContext заголовок (`traceparent`). В Spring Boot + Micrometer Tracing trace ID автоматически добавляется в MDC и лог. Grafana позволяет из dashboard перейти по trace ID к Loki-логам и Jaeger-трейсам.

## Logging (Q25–Q29)

### Q25: Почему структурированное логирование лучше строк?
**A:** Строковые логи парсятся regex — хрупко и медленно. JSON логи легко индексируются и фильтруются: `{level: "ERROR", userId: "123", orderId: "456"}`. В Loki/Elasticsearch можно делать `userId = "123"` вместо regex. Структурированные поля позволяют строить агрегации: "все ошибки по userId за последний час". На практике это разница между "grep в панике" и "SQL-запрос к логам".

### Q26: Что такое MDC? Как работает с async?
**A:** MDC (Mapped Diagnostic Context) — thread-local Map, автоматически добавляемая к каждой записи в лог. Пример: `MDC.put("requestId", uuid)` → все последующие `log.info(...)` в том же потоке будут содержать `requestId`. Проблема с async: при передаче задачи в другой поток MDC не копируется автоматически. Решение: `TaskDecorator` в Spring, который перед выполнением задачи копирует MDC из родительского потока, а после очищает.

### Q27: Что логировать на уровне ERROR vs WARN?
**A:** `ERROR` — ситуация, требующая немедленного внимания: неожиданное исключение, недоступность внешней зависимости, нарушение инварианта данных. На ERROR должен срабатывать алерт. `WARN` — нештатная, но обрабатываемая ситуация: retry успешен после 2 попыток, запрос упал на rate limit, deprecated API. `INFO` — нормальные бизнес-события: заказ создан, платёж прошёл. `DEBUG` — детали реализации, не нужные в production.

### Q28: Как реализовать correlation ID в Spring Boot?
**A:** Через `OncePerRequestFilter`: если в запросе есть заголовок `X-Request-ID` — берём его, иначе генерируем UUID. Кладём в MDC: `MDC.put("requestId", id)`. Добавляем заголовок в ответ. В `finally` блоке: `MDC.remove("requestId")`. Если сервис делает HTTP-вызовы downstream — нужно прокидывать ID через `RestTemplate`/`WebClient` interceptor. Для Kafka — через message header.

### Q29: Что такое антипаттерн "log and throw"?
**A:** Логирование исключения и одновременное его перебрасывание. Это приводит к дублированию одного и того же exception stacktrace несколько раз в логах — на каждом уровне вызовов. Правильно: либо логировать и обрабатывать (`catch + log`), либо пробрасывать без логирования (`throw`). Логировать нужно только один раз — там, где exception обрабатывается окончательно (обычно на уровне верхнего exception handler).

## Metrics (Q30–Q36)

### Q30: Чем Counter отличается от Gauge?
**A:** Counter — монотонно возрастающее значение (только увеличивается), например, количество обработанных запросов. Никогда не уменьшается, только сбрасывается при рестарте. Для анализа используют `rate()` или `increase()`. Gauge — текущее значение, которое может расти и убывать: размер очереди, количество активных соединений, использование памяти. Gauge читается напрямую без `rate()`.

### Q31: Когда использовать Histogram, а когда Summary?
**A:** Histogram считает наблюдения по bucket'ам (предопределённые границы, например `[0.1, 0.5, 1, 5]` секунд) на стороне приложения и позволяет агрегировать percentile в Prometheus через `histogram_quantile()` — работает при нескольких инстансах приложения. Summary считает percentile в самом приложении — не агрегируется между инстансами. Правило: почти всегда используйте Histogram. Summary имеет смысл только если percentile нужны сразу без PromQL и инстанс один.

### Q32: Как работает pull-модель Prometheus?
**A:** Prometheus сам ходит к каждому приложению по расписанию (scrape interval, обычно 15s) и забирает метрики с endpoint `/metrics`. Приложение должно предоставить endpoint в формате Prometheus text exposition. В Spring Boot это `/actuator/prometheus` при наличии `micrometer-registry-prometheus`. Альтернатива — push через Pushgateway (для batch jobs, которые живут меньше scrape interval). Pull проще для service discovery и гарантирует что мертвые сервисы не шлют метрики.

### Q33: Что такое `rate()` в PromQL? Почему нельзя применять к Gauge?
**A:** `rate(counter[5m])` вычисляет среднюю скорость изменения counter за 5 минут (per second). Корректно обрабатывает resets (перезапуск приложения). `irate()` — то же, но по двум последним точкам (реагирует быстрее). К Gauge применять `rate()` нельзя: Gauge может убывать, а `rate()` предполагает что значение только растёт и обрабатывает "уменьшение" как reset. Для Gauge используют прямое значение, `delta()` или `deriv()`.

### Q34: Как настроить метрики в Spring Boot?
**A:** Добавить `micrometer-registry-prometheus` в зависимости. В `application.properties`: `management.endpoints.web.exposure.include=prometheus`. Метрики доступны на `/actuator/prometheus`. Для кастомных метрик: инжектировать `MeterRegistry` и использовать `registry.counter("orders.created", "status", status).increment()`. Аннотация `@Timed` на методе автоматически создаёт Timer. Spring Boot автоматически регистрирует метрики JVM, HTTP, DataSource.

### Q35: Что такое кардинальность (cardinality) в контексте метрик? Почему важно?
**A:** Кардинальность — количество уникальных комбинаций label. `http_requests_total{method, status}` с 5 методами × 10 статусов = 50 time series. Если добавить `userId` (миллион пользователей) — получим 50M time series. Высокая кардинальность убивает Prometheus: память, disk, время запросов. Правило: не использовать в label значения с неограниченным множеством (user ID, email, URL без нормализации). Группируйте: вместо конкретного status code — `2xx`, `4xx`, `5xx`.

### Q36: Как `histogram_quantile()` вычисляет p99?
**A:** Histogram хранит counts по bucket'ам: сколько запросов выполнились быстрее 0.1с, 0.5с, 1с, и т.д. `histogram_quantile(0.99, rate(http_request_duration_seconds_bucket[5m]))` интерполирует внутри bucket, в котором находится 99-я перцентиль. Точность зависит от bucket'ов: если 99-я перцентиль между 0.5с и 1с, но bucket'ов между ними нет — будет приближение. Для Spring Boot/Micrometer bucket'ы настраиваются через `management.metrics.distribution.sla.*`.
